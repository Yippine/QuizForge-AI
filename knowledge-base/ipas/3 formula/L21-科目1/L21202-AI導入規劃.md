# L21202 - AI導入規劃

## 1. 核心定義 (20%, 400-1200字)

### 1.1 主題定義

AI導入規劃是在通過可行性評估後,針對AI專案的範疇、資源、時程、團隊、技術選型、風險管理等面向,制定系統化執行計畫的過程。其核心目標是將抽象的AI願景轉化為具體可執行的專案藍圖,確保資源有效配置、風險可控、進度可追蹤,最終實現預期業務價值。

AI導入規劃不僅是技術實施計畫,更是商業策略落地的關鍵環節。規劃需兼顧技術可行性與業務價值,在不確定性中建立清晰的執行路徑,協調跨部門協作,確保專案在預算、時程、品質三重限制下成功交付。

### 1.2 核心概念

**AI導入規劃的五大核心概念:**

1. **專案範疇管理(Scope Management)**:定義專案邊界、交付物、驗收標準,避免範疇蔓延
2. **資源配置策略(Resource Allocation)**:人力、資料、運算、預算等資源的最佳化分配
3. **時程規劃(Schedule Planning)**:里程碑設定、關鍵路徑分析、緩衝時間管理
4. **團隊組建(Team Formation)**:角色定義、技能矩陣、內外部資源整合
5. **技術架構設計(Technical Architecture)**:模型選型、系統架構、資料流程、整合策略

### 1.3 CFDS 分解

基於 Formula-Contract 方法論,將AI導入規劃過程分解為四個基本單元:

```
AI_Planning = f(C, F, D, S)
```

**C (Code - 可執行邏輯)**
```
C = ScopeDefinition ∘ ResourcePlanning ∘ ScheduleDesign ∘ TeamBuilding ∘ RiskPlanning ∘ ExecutionPreparation
  = 範疇定義 -> 資源規劃 -> 時程設計 -> 團隊組建 -> 風險規劃 -> 執行準備
```

**F (Files - 配置資源)**
```
F = {專案章程, WBS工作分解, 甘特圖, 資源矩陣, 風險登記簿, 溝通計畫, 變更管理流程}
```

**D (Data - 資料結構)**
```
D = ProjectScope + ResourcePlan + Schedule + TeamStructure + RiskRegister + BudgetForecast
  = 專案範疇 + 資源計畫 + 時程表 + 團隊結構 + 風險登記 + 預算預測
```

**S (State - 運行狀態)**
```
S = Planning | Approval | Execution | Monitoring | Closure
  = 規劃中 | 審批中 | 執行中 | 監控中 | 結案
```

### 1.4 技術定位

AI導入規劃在專案生命週期中處於「規劃階段」,是連接「評估決策」與「執行實施」的橋樑。規劃品質直接影響專案執行效率與成功率,是專案管理的核心環節。

在AI治理框架中,導入規劃是將策略轉化為行動的關鍵,涉及技術路線、組織變革、流程優化的綜合設計,需要專案管理、技術架構、業務領域的跨領域整合能力。

## 2. 關鍵公式 (25%, 500-1500字)

### 2.1 AI專案規劃核心公式

```
AI_Planning = ScopeManagement + ResourceAllocation + SchedulePlanning + TeamBuilding + RiskManagement

詳細展開:
  ScopeManagement = Objectives + Deliverables + Acceptance Criteria + Exclusions
  ResourceAllocation = HumanResources × ComputeResources × DataResources × Budget
  SchedulePlanning = WBS ∘ Dependencies ∘ CriticalPath ∘ Milestones
  TeamBuilding = RoleDefinition + SkillMapping + Recruitment/Training
  RiskManagement = RiskIdentification + Mitigation Planning + Contingency Reserve
```

### 2.2 工作分解結構(WBS)公式

```
WBS = DecomposeProject -> Tasks -> SubTasks -> WorkPackages

AI專案典型WBS:
  L1: AI專案
  L2: Phase1(資料準備) + Phase2(模型開發) + Phase3(系統整合) + Phase4(部署上線)
  L3: Phase1.1(資料收集) + Phase1.2(資料清理) + Phase1.3(資料標註) + Phase1.4(資料驗證)
  L4: Phase1.1.1(需求分析) + Phase1.1.2(資料來源識別) + Phase1.1.3(資料擷取) + Phase1.1.4(品質檢查)

任務估算:
  TaskEffort = Optimistic + 4×MostLikely + Pessimistic / 6 (PERT三點估算)
  TaskDuration = TaskEffort / ResourceAvailability
```

### 2.3 關鍵路徑法(CPM)公式

```
CriticalPath = LongestPath(Start -> End) with Zero Float

計算步驟:
  1. 繪製網路圖(Network Diagram)
  2. 計算最早開始/完成時間(ES/EF): Forward Pass
     EF = ES + Duration
  3. 計算最晚開始/完成時間(LS/LF): Backward Pass
     LS = LF - Duration
  4. 計算浮動時間(Float): Float = LF - EF = LS - ES
  5. 識別關鍵路徑: Float = 0 的任務鏈

專案總工期 = CriticalPath Duration
```

### 2.4 資源平衡公式

```
ResourceLeveling = AdjustSchedule(ResourceDemand, ResourceCapacity)

資源需求計算:
  Daily_Demand_t = Σ(Tasks active on day t) × Resource_Requirement_per_Task

資源超載檢測:
  if Daily_Demand_t > Resource_Capacity_t:
    解決方案: 延後非關鍵任務 | 增加資源 | 調整任務範疇

資源利用率:
  Utilization = Actual_ResourceUsed / Total_ResourceAvailable × 100%
  目標: 70-85% (過低浪費,過高風險)
```

### 2.5 預算估算公式

```
ProjectBudget = DirectCosts + IndirectCosts + ContingencyReserve + ManagementReserve

AI專案成本結構:
  DirectCosts = PersonnelCosts + ComputeInfrastructure + Software/ToolsLicenses + DataAcquisition + Consulting
  IndirectCosts = Overhead + Administration + Facility
  ContingencyReserve = IdentifiedRisks × Probability × Impact (已知風險)
  ManagementReserve = TotalCost × 5-10% (未知風險)

成本估算方法:
  類比估算: 參考類似專案歷史成本
  參數估算: Cost = Unit_Rate × Quantity (如:工程師日薪 × 人天)
  自下而上: Σ(WBS各任務成本)
```

### 2.6 團隊組建公式

```
TeamStructure = RoleDefinition + SkillMatrix + Responsibility Assignment

典型AI專案角色:
  BusinessSide = ProductOwner + BusinessAnalyst + DomainExpert
  TechnicalSide = DataScientist + DataEngineer + MLEngineer + SoftwareEngineer + DevOpsEngineer
  Management = ProjectManager + ScrumMaster
  Support = QA Engineer + UX Designer + Legal/Compliance

RACI矩陣(責任分配):
  R (Responsible): 執行者
  A (Accountable): 當責者(唯一,做決策)
  C (Consulted): 諮詢者
  I (Informed): 知情者

技能缺口分析:
  Skill_Gap = Required_Skills - Current_Skills
  填補策略: Training | Recruitment | Outsourcing | Partnership
```

### 2.7 敏捷規劃公式

```
AgilePlanning = ProductBacklog + SprintPlanning + Velocity Tracking

Sprint規劃:
  Sprint_Capacity = Team_Size × Working_Days × Focus_Factor
    Focus_Factor = 0.6-0.8 (考慮會議、支援等非開發時間)

  Story_Points_per_Sprint = Team_Velocity (歷史平均值)

優先順序排序(MoSCoW):
  Must have: 核心功能,MVP必備
  Should have: 重要但非必要
  Could have: 有加分但可延後
  Won't have: 本版本不做

發布規劃:
  Release_Date = Start_Date + (Total_Story_Points / Team_Velocity) × Sprint_Length
```

## 4. 實務應用 (20%, 400-1200字)

### 4.1 應用場景一:電商推薦系統導入規劃

**場景描述**:電商平台已通過評估,決定導入AI個性化推薦系統,需制定詳細執行計畫。

**規劃流程:**
```
Step1: 範疇定義
  目標: 提升點擊率3倍(2%->6%),轉換率2倍(0.5%->1%)
  交付物: 推薦引擎API、A/B測試平台、監控儀表板
  範疇邊界: 僅商品推薦,不含廣告推薦與內容推薦
  驗收標準: 離線AUC>0.85, 線上點擊率>5%, P95延遲<100ms

Step2: WBS工作分解
  Phase1: 資料準備(4週)
    1.1 使用者行為日誌收集與清理(2週)
    1.2 商品特徵工程(1週)
    1.3 訓練/驗證/測試集切分(1週)
  
  Phase2: 模型開發(6週)
    2.1 基線模型開發(協同過濾)(2週)
    2.2 深度學習模型開發(DeepFM)(3週)
    2.3 模型調優與驗證(1週)
  
  Phase3: 系統整合(4週)
    3.1 推薦引擎API開發(2週)
    3.2 A/B測試框架建置(1週)
    3.3 整合測試(1週)
  
  Phase4: 部署上線(2週)
    4.1 生產環境部署(1週)
    4.2 監控與優化(1週)

Step3: 資源配置
  團隊: 
    資料科學家 × 2(全職,16週)
    資料工程師 × 1(全職,8週)
    後端工程師 × 1(全職,6週)
    專案經理 × 1(兼職,16週)
  
  運算資源:
    訓練: GPU伺服器(V100 × 4, 6週)
    推理: CPU伺服器(16核 × 4台)
  
  預算:
    人力: 400萬(8人月 × 50萬)
    硬體: 200萬(租賃費用)
    軟體授權: 50萬
    緩衝: 100萬(15%)
    總計: 750萬

Step4: 時程規劃
  總工期: 16週(4個月)
  關鍵路徑: 資料準備 -> 深度學習模型開發 -> API開發 -> 部署
  里程碑:
    M1(週4): 資料集準備完成
    M2(週10): 模型達標(AUC>0.85)
    M3(週14): 系統整合測試通過
    M4(週16): 正式上線

Step5: 風險規劃
  風險1(高): 資料品質不佳導致模型準確度不足
    緩解: 提前進行資料稽核,預留2週緩衝
  風險2(中): 深度學習模型訓練時間超出預期
    緩解: 準備基線模型作為備案,雲端GPU預算增加20%
  風險3(中): API整合遇到效能瓶頸
    緩解: 預先進行效能測試,準備快取與負載平衡方案
```

### 4.4 規劃實作步驟(通用流程)

**AI專案規劃八步驟:**
```
Step1: 專案啟動會議(Kick-off)
  明確目標、範疇、利害關係人期待

Step2: WBS工作分解
  將專案拆解為可管理的任務包

Step3: 任務估算與排序
  PERT三點估算,識別依賴關係

Step4: 資源規劃
  人力、運算、資料、預算配置

Step5: 時程規劃
  甘特圖、關鍵路徑、里程碑設定

Step6: 風險規劃
  識別、評估、緩解、監控

Step7: 溝通計畫
  利害關係人溝通頻率與方式

Step8: 計畫審查與核准
  管理層審查並核准執行計畫
```

### 4.5 常見陷阱與解決方案

**陷阱1:範疇蔓延(Scope Creep)**
- 問題:專案執行中不斷增加需求,導致時程與預算超支
- 解決:嚴格變更管理流程,所有新需求需評估影響並取得核准

**陷阱2:過度樂觀的時程估算**
- 問題:低估AI專案的不確定性與複雜度
- 解決:PERT三點估算+20-30%緩衝時間+參考歷史專案數據

**陷阱3:技能缺口未提前規劃**
- 問題:專案執行時才發現團隊缺乏關鍵技能
- 解決:技能矩陣分析+提前招募/訓練+外部顧問備案

**陷阱4:關鍵路徑忽略**
- 問題:非關鍵任務延誤被忽視,最終影響交付
- 解決:定期檢視關鍵路徑,優先資源配置給關鍵任務

**陷阱5:利害關係人溝通不足**
- 問題:專案進度與期待不一致,最終驗收困難
- 解決:定期進度報告+Demo展示+建立溝通節奏

```
想像專案是一場戲:
R (Responsible) = 演員(實際演出)
A (Accountable) = 導演(做決策,唯一)
C (Consulted) = 編劇(諮詢意見)
I (Informed) = 觀眾(知道結果)

一個任務可以多個R,但A只能一個(避免責任分散)
```

**關鍵路徑記憶法(項鍊理論):**
```
想像專案任務是串珠項鍊:
每顆珠 = 一個任務
項鍊長度 = 專案總工期

關鍵路徑 = 最長的那條項鍊
浮動時間 = 較短項鍊的鬆弛空間

拉長關鍵路徑上任一顆珠 -> 整個項鍊變長(專案延誤)
拉長非關鍵路徑的珠 -> 只要在浮動時間內,項鍊不變
```

### 5.3 快速回憶提示

**考試快速回想關鍵字:**
- 規劃要素 = 範疇+資源+時程+團隊+風險
- WBS = 階段->任務->子任務->工作包
- 關鍵路徑 = Float = 0的任務鏈
- RACI = R執行,A當責,C諮詢,I知情
- 敏捷 = Product Backlog + Sprint + Demo

### 5.4 易混淆辨析

**PoC vs Pilot vs Production:**
- **PoC(概念驗證)**:實驗室環境,驗證技術可行,小規模
- **Pilot(試點)**:真實環境,小範圍部署,驗證業務價值
- **Production(生產)**:全面推廣,規模化部署
- 記憶:PoC是「試吃」,Pilot是「試營運」,Production是「正式開業」

**甘特圖 vs 看板(Kanban):**
- **甘特圖**:瀑布式,時間軸視圖,適合預先規劃
- **看板**:敏捷式,工作流視圖,適合動態調整
- 記憶:甘特圖是「時刻表」(固定班次),看板是「流水線」(持續流動)

**直接成本 vs 間接成本:**
- **直接成本**:可直接歸屬專案(人力、硬體、軟體)
- **間接成本**:無法直接歸屬,需分攤(管理費、辦公室租金)
- 記憶:直接是「專屬」,間接是「共享」

### 6.2 簡答題

**題目6**:為什麼AI專案適合採用「混合式」(瀑布+敏捷)專案管理方法?請說明原因與實施建議。

**答案解析**:
- **適合混合式的原因**:
  1. **不確定性混合**:資料準備與系統整合較確定(適合瀑布),模型開發需實驗迭代(適合敏捷)
  2. **利害關係人需求**:管理層需要明確時程與預算(瀑布),開發團隊需要彈性調整(敏捷)
  3. **風險管理**:關鍵里程碑用瀑布管控,內部迭代用敏捷加速
  4. **法規合規**:某些文件與審查流程需瀑布式嚴謹管控

- **實施建議**:
  1. **階段劃分**:
     - 瀑布式:資料準備、系統架構設計、部署上線
     - 敏捷式:模型開發、特徵工程、超參數調優
  2. **里程碑固定,內部彈性**:
     - 對外承諾關鍵里程碑(如M1:資料就緒,M2:模型達標)
     - 內部採用Sprint迭代,快速試錯
  3. **文件適度**:
     - 關鍵決策與架構文件需正式(瀑布)
     - 開發細節用輕量化文件(敏捷)
  4. **溝通分層**:
     - 對管理層:每月正式報告(瀑布)
     - 團隊內部:每日站立會議+Sprint Review(敏捷)

**題目7**:請說明如何進行AI專案的資源平衡(Resource Leveling),並舉例說明資源衝突的解決策略。

**答案解析**:
- **資源平衡定義**:調整專案時程,使資源需求不超過可用資源,避免資源超載或浪費

- **步驟**:
  1. **繪製資源負荷圖**:橫軸=時間,縱軸=資源需求,繪製每日/週資源需求曲線
  2. **識別超載點**:找出資源需求 > 可用資源的時段
  3. **調整策略**:
     - 延後非關鍵任務:利用浮動時間,將任務往後挪
     - 拆分任務:將大任務分階段執行
     - 增加資源:招募或外包(增加成本)
     - 縮小範疇:移除非必要功能

- **範例**:
  ```
  情境:資料科學家團隊3人,同時期有3個任務:
    Task A(關鍵路徑): 資料清理,需2人,2週
    Task B(Float=1週): 特徵工程,需2人,1週
    Task C(Float=2週): EDA探索,需1人,1週

  資源衝突:第1-2週需要5人,但只有3人

  解決方案:
    Option 1: 延後Task B至第3週(利用Float)
      第1-2週: Task A(2人) + Task C(1人) = 3人 ✓
      第3週: Task B(2人)
    
    Option 2: Task A與Task B分段執行
      第1週: Task A第一階段(2人) + Task C(1人)
      第2週: Task B(2人) + Task A第二階段(1人)
    
    建議:選Option 1,保持關鍵路徑任務連續性,減少上下文切換成本
  ```

### 6.3 易錯點提醒

**易錯點1**:混淆WBS與甘特圖
- **正確理解**:WBS是工作分解結構(層次圖),甘特圖是時程視圖(時間軸)。WBS先做(定義做什麼),甘特圖後做(規劃何時做)
- **記憶提示**:WBS是「地圖」(空間),甘特圖是「行程表」(時間)

**易錯點2**:認為關鍵路徑不會改變
- **正確理解**:專案執行中,若非關鍵任務大幅延誤超過Float,可能變成新的關鍵路徑,需動態監控
- **記憶提示**:關鍵路徑是「動態」的,非「靜態」

**易錯點3**:預算只估直接成本,忽略間接成本與緩衝
- **正確理解**:完整預算 = 直接成本 + 間接成本 + 已知風險緩衝(5-10%) + 未知風險緩衝(5-10%)
- **記憶提示**:預算要「全面」+「緩衝」,避免超支

**易錯點4**:團隊技能缺口評估過於樂觀
- **正確理解**:AI專案需跨領域技能(資料、演算法、工程、業務),需客觀評估現有團隊能力,提前規劃訓練或招募
- **記憶提示**:技能評估要「實事求是」,避免「理想化」
