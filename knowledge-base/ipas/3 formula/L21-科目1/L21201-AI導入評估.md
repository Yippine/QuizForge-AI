# L21201 - AI導入評估

## 1. 核心定義 (20%, 400-1200字)

### 1.1 主題定義

AI導入評估是企業在正式啟動AI專案前,針對業務需求、技術條件、成本效益、組織能力等多面向進行系統性分析與判斷的過程。其核心目標是確保AI專案具備「必要性」與「可能性」,避免資源浪費與決策失誤,為後續的規劃與實施建立穩健基礎。

AI導入評估不僅聚焦於技術是否可實現,更需關注其是否能解決企業當前痛點、與業務目標契合,並具備合理的成本效益。透過結構化的評估過程,企業能清晰判斷AI專案的可行性、優先順序與預期價值,確保投資決策建立在充分資訊與理性分析之上。

### 1.2 核心概念

**AI導入評估的五大核心概念:**

1. **業務需求分析(Business Need Analysis)**:識別企業痛點、定義問題範疇、評估AI解決方案適配性
2. **技術可行性評估(Technical Feasibility)**:檢視資料品質、技術成熟度、系統相容性、團隊能力
3. **成本效益分析(Cost-Benefit Analysis)**:量化投資成本與預期收益,計算ROI、NPV、回收期
4. **風險評估(Risk Assessment)**:識別技術、資料、組織、法規等面向風險,制定緩解策略
5. **組織準備度(Organizational Readiness)**:評估組織文化、人才結構、流程配合度、變革管理能力

### 1.3 CFDS 分解

基於 Formula-Contract 方法論,將AI導入評估過程分解為四個基本單元:

```
AI_Assessment = f(C, F, D, S)
```

**C (Code - 可執行邏輯)**
```
C = NeedIdentification ∘ FeasibilityAnalysis ∘ CostBenefitCalculation ∘ RiskEvaluation ∘ DecisionMaking
  = 需求識別 -> 可行性分析 -> 成本效益計算 -> 風險評估 -> 決策制定
```

**F (Files - 配置資源)**
```
F = {評估範本, 財務模型, 風險矩陣, 技術檢核表, 組織能力評量表, 決策樹}
```

**D (Data - 資料結構)**
```
D = BusinessRequirements + TechnicalSpecs + CostData + BenefitProjections + RiskRegistry
  = 業務需求 + 技術規格 + 成本數據 + 效益預估 + 風險清單
```

**S (State - 運行狀態)**
```
S = InitialScoping | DetailedAssessment | StakeholderReview | ApprovalDecision
  = 初步界定 | 詳細評估 | 利害關係人審查 | 核准決策
```

### 1.4 技術定位

AI導入評估在AI專案生命週期中處於「啟動階段」的關鍵位置,是從「構想」邁向「執行」的決策閘門(Decision Gate)。評估品質直接影響後續專案成敗,是AI治理(AI Governance)的重要環節。

在企業AI轉型中,導入評估扮演資源配置與策略對齊的角色,協助管理層識別高價值機會、排定優先順序、控制風險,確保AI投資與商業目標一致。

## 2. 關鍵公式 (25%, 500-1500字)

### 2.1 AI導入評估核心流程公式

```
AI_Assessment = NeedAnalysis -> FeasibilityCheck -> CostBenefit -> RiskAssessment -> GoNoGo

詳細展開:
  NeedAnalysis = ProblemDefinition + StakeholderAlignment + ObjectiveSetting
  FeasibilityCheck = DataQuality × TechnicalMaturity × SystemCompatibility × TeamCapability
  CostBenefit = (Benefits - Costs) / Costs -> ROI_Decision
  RiskAssessment = RiskIdentification + RiskQuantification + MitigationPlanning
  GoNoGo = (Feasibility × ROI × Risk_Acceptable) -> Approve | Defer | Reject
```

### 2.2 成本效益分析公式

**投資報酬率(ROI):**
```
ROI = (投資回報 - 投資成本) / 投資成本 × 100%

其中:
  投資成本 = 初期開發成本 + 年度營運成本 × 年數
           = (人力 + 硬體 + 軟體 + 訓練 + 顧問) + (維護 + 更新 + 授權)

  投資回報 = 效率提升收益 + 成本節省 + 營收增長 + 風險降低價值
           = Σ(年度效益) (累計至評估期間)

判斷標準:
  ROI > 50% (1年內) -> 高價值專案
  ROI > 100% (2年內) -> 中價值專案
  ROI < 50% (2年內) -> 低價值/高風險專案
```

**淨現值(NPV - Net Present Value):**
```
NPV = Σ(t=1 to n) [Cash_Flow_t / (1 + r)^t] - Initial_Investment

其中:
  Cash_Flow_t = 第t年的淨現金流(收益-成本)
  r = 折現率(Discount Rate),通常為企業加權平均資本成本(WACC)
  n = 專案生命週期(年數)

決策準則:
  NPV > 0 -> 專案具經濟價值,建議執行
  NPV = 0 -> 專案收益等於成本,中性
  NPV < 0 -> 專案不具經濟價值,不建議執行
```

**回收期(Payback Period):**
```
Payback Period = Initial_Investment / Annual_Net_Cash_Flow

範例:
  初期投資 = 500萬
  年度淨收益 = 150萬
  回收期 = 500 / 150 = 3.33年

一般標準:
  < 1年 -> 極佳
  1-2年 -> 優良
  2-3年 -> 可接受
  > 3年 -> 需謹慎評估
```

### 2.3 技術可行性評估公式

**資料就緒度評分(Data Readiness Level):**
```
Data_Readiness = Quality × Quantity × Accessibility × Relevance

細項評分(0-5分):
  Quality = Completeness(完整性) + Accuracy(準確性) + Consistency(一致性)
  Quantity = Volume(數量充足) + Diversity(多樣性)
  Accessibility = Availability(可取得) + Format(格式適用)
  Relevance = Business_Alignment(業務相關) + Feature_Richness(特徵豐富)

總分判斷:
  80-100分 -> 資料就緒,可直接開發
  60-79分 -> 需適度清理與增強
  40-59分 -> 需大量資料工程
  < 40分 -> 資料不足,需重新收集
```

**技術成熟度評估(Technology Readiness Level, TRL):**
```
TRL_Score = ModelMaturity + InfrastructureReadiness + IntegrationComplexity

分級標準(1-9級):
  TRL 1-3: 概念驗證階段(Proof of Concept)
  TRL 4-6: 技術驗證與試點(Pilot)
  TRL 7-9: 系統整合與規模部署(Production)

AI專案建議:
  選擇 TRL 7-9 的成熟技術 -> 風險低,成功率高
  選擇 TRL 4-6 的技術 -> 需投入研發資源
  選擇 TRL 1-3 的技術 -> 高風險,適合創新專案
```

### 2.4 風險評估矩陣公式

**風險評分公式:**
```
Risk_Score = Probability(機率) × Impact(影響)

機率分級(1-5):
  1 = 極低(< 10%)
  2 = 低(10-30%)
  3 = 中(30-50%)
  4 = 高(50-70%)
  5 = 極高(> 70%)

影響分級(1-5):
  1 = 微小(可忽略)
  2 = 輕微(輕微延誤/成本)
  3 = 中度(顯著影響)
  4 = 嚴重(專案受阻)
  5 = 災難(專案失敗)

風險矩陣:
  Score 1-5: 低風險(綠燈) -> 接受
  Score 6-12: 中風險(黃燈) -> 監控與緩解
  Score 13-20: 高風險(橙燈) -> 強化緩解措施
  Score 21-25: 極高風險(紅燈) -> 考慮終止或重新設計
```

### 2.5 業務價值優先順序評估

**價值優先矩陣:**
```
Priority_Score = (BusinessValue × Feasibility) / (Cost × Risk)

其中:
  BusinessValue = Strategic_Alignment + Revenue_Impact + Cost_Saving + Competitive_Advantage
  Feasibility = Technical_Feasibility + Data_Readiness + Organizational_Readiness
  Cost = Development_Cost + Operational_Cost + Opportunity_Cost
  Risk = Technical_Risk + Business_Risk + Compliance_Risk

優先順序排序:
  Priority_Score > 5 -> 高優先級(Quick Win)
  Priority_Score 2-5 -> 中優先級(Strategic)
  Priority_Score < 2 -> 低優先級(Consider Later)

2×2矩陣決策:
  高價值 × 高可行性 -> 立即執行(Quick Wins)
  高價值 × 低可行性 -> 策略專案(需投入資源)
  低價值 × 高可行性 -> 填補專案(有餘力再做)
  低價值 × 低可行性 -> 放棄(資源錯配)
```

### 2.6 AI適用性判斷公式

**AI Solution Fit Score:**
```
AI_Fit = (Problem_Complexity + Data_Availability + Automation_Benefit) - Traditional_Solution_Effectiveness

判斷準則:
  AI_Fit > 7 -> 高度適合AI解決
  AI_Fit 4-7 -> 適合AI,但需評估成本
  AI_Fit < 4 -> 傳統方法可能更合適

問題特徵(適合AI):
  - 規則複雜難以窮舉
  - 需要處理大量資料
  - 存在模式識別需求
  - 需要個性化與適應性
  - 人工作業重複性高且耗時

問題特徵(不適合AI):
  - 規則明確且穩定
  - 資料量小且結構化
  - 可用確定性演算法解決
  - 法規要求100%可解釋性
  - 錯誤容忍度為零
```

| 評估方法 | 適用階段 | 時間成本 | 精確度 | 適用專案規模 | 主要產出 |
|---------|---------|---------|--------|-------------|---------|

### 3.3 AI專案 vs 傳統IT專案評估差異

| 評估項目 | 傳統IT專案 | AI專案 |
|---------|-----------|--------|
| **需求明確性** | 需求明確、規格固定 | 需求可能在開發中調整 |
| **資料依賴度** | 低,主要是交易資料 | 高,需大量高品質訓練資料 |
| **成功標準** | 功能完成度、系統穩定性 | 模型準確度、業務指標提升 |
| **風險特性** | 技術風險為主 | 資料、模型、業務多重風險 |
| **投資回報** | 較易量化 | 部分效益難以精確量化 |
| **評估重點** | 功能、架構、成本 | 資料、模型、適用性、倫理 |

### 3.4 財務指標對比

| 財務指標 | 計算方式 | 優點 | 缺點 | 適用情境 |
|---------|---------|------|------|---------|
| **ROI** | (回報-成本)/成本 | 直觀易懂 | 忽略時間價值 | 快速評估 |
| **NPV** | 折現現金流總和 | 考慮時間價值 | 需準確估計折現率 | 長期專案 |
| **IRR** | 使NPV=0的折現率 | 與企業資本成本比較 | 計算複雜 | 多專案比較 |
| **回收期** | 成本/年度收益 | 簡單明瞭 | 忽略回收期後收益 | 風險敏感專案 |

### 3.5 決策框架對比

```
評估結果決策樹:

評估結果
├─ 業務需求不明確?
│  └─ Yes -> 退回需求分析階段,重新定義問題
├─ 技術可行性不足?
│  ├─ 資料問題 -> PoC驗證 or 資料收集計畫
│  └─ 技術問題 -> 尋找替代方案 or 外部合作
├─ 成本效益不佳?
│  ├─ ROI < 閾值 -> 縮小範疇 or 延後執行
│  └─ 回收期過長 -> 分階段實施 or 尋求補助
├─ 風險過高?
│  ├─ 可緩解 -> 制定風險管理計畫
│  └─ 不可控 -> 考慮終止專案
└─ 全面通過?
   └─ 進入規劃階段,制定詳細實施計畫
```

## 4. 實務應用 (20%, 400-1200字)

### 4.1 應用場景一:製造業品質檢測AI導入評估

**場景描述**:傳統製造廠考慮導入AI視覺檢測,取代人工品檢,降低漏檢率與人力成本。

**評估流程:**
```
Step1: 業務需求分析
  痛點: 人工品檢漏檢率5%,年損失500萬;品檢人力成本600萬/年
  目標: 漏檢率降至1%,人力成本降低50%
  預期效益: 年節省成本350萬(損失減少400萬+人力減少300萬-新增成本350萬)

Step2: 技術可行性檢查
  資料: 已有10萬張標註瑕疵影像
  技術: CNN物件偵測技術成熟(TRL 8)
  系統: 現有產線可加裝攝影機模組
  團隊: 需外部顧問協助,內部培養維運人員

Step3: 成本效益計算
  初期成本: 硬體200萬 + 軟體開發150萬 + 顧問100萬 = 450萬
  年度成本: 維護50萬 + 授權30萬 = 80萬
  ROI(第1年) = (350-80-450)/450 = -40% (負值,首年未回收)
  ROI(第2年累計) = (350×2-80×2-450)/450 = 20%
  回收期 = 450/(350-80) = 1.67年

Step4: 風險評估
  技術風險(中): 新型瑕疵可能辨識率低 -> 持續訓練機制
  業務風險(低): 誤判導致良品報廢 -> 人工複檢機制
  組織風險(中): 品檢人員抗拒 -> 轉崗訓練計畫

Step5: 決策建議
  建議執行,但需:
  - 分階段導入(先試點單一產線)
  - 建立人機協作機制(AI輔助,人工覆核)
  - 制定員工轉型計畫
```

### 4.2 應用場景二:金融業客戶流失預測評估

**場景描述**:銀行希望導入AI預測客戶流失,提前挽留高價值客戶。

**評估流程:**
```
Step1: 業務需求
  痛點: 年客戶流失率15%,高價值客戶流失造成年損3000萬
  目標: 提前3個月預測流失,挽留率提升30%
  預期效益: 年增營收900萬(3000萬×30%)

Step2: 技術可行性
  資料: 5年客戶交易與行為資料,樣本50萬筆
  資料品質: 需處理缺失值(10%)與資料不平衡(流失客戶僅15%)
  技術: 分類模型(Random Forest/XGBoost) TRL 9
  整合: 需與CRM系統整合,觸發行銷活動

Step3: 成本效益
  開發成本: 200萬(含資料工程)
  行銷成本: 挽留活動預算300萬/年
  ROI = (900-300-200)/200 = 200% (第1年)
  NPV(3年,折現率10%) = 900/1.1 + 900/1.1² + 900/1.1³ - 500 = 1738萬

Step4: 風險評估
  模型風險(中): 過度擬合訓練資料 -> 交叉驗證+定期重訓
  隱私風險(高): 客戶資料使用需符合個資法 -> 法律審查+去識別化
  業務風險(低): 過度行銷騷擾客戶 -> 設定接觸頻率上限

Step5: 決策建議
  高度建議執行,ROI與NPV皆優異,但需:
  - 強化資料治理與隱私保護
  - 設計客戶友善的挽留策略
  - 建立模型監控機制
```

### 4.3 應用場景三:零售業智慧推薦系統評估

**場景描述**:電商平台評估導入個性化商品推薦AI,提升轉換率。

**評估流程:**
```
Step1: 業務需求
  痛點: 現有規則式推薦點擊率僅2%,轉換率0.5%
  目標: 點擊率提升至5%,轉換率提升至1.5%
  預期效益: 年增營收5000萬(平均客單價3000元,轉換提升帶來167萬筆訂單)

Step2: 技術可行性
  資料: 億級使用者行為日誌,商品特徵完整
  技術: 協同過濾+深度學習推薦模型(TRL 8)
  系統: 需建置實時推薦引擎,挑戰大
  團隊: 需建立演算法團隊(3-5人)

Step3: 成本效益
  初期: 演算法開發300萬 + 基礎建設500萬 = 800萬
  年度: 人力成本400萬 + 運算成本200萬 = 600萬
  ROI(第1年) = (5000-600-800)/800 = 462.5%
  回收期 = 800/(5000-600) = 0.18年(2個月)

Step4: 風險評估
  技術風險(中): 冷啟動問題(新用戶無歷史) -> 內容式推薦補充
  業務風險(低): 推薦同質化降低探索 -> 引入探索機制
  倫理風險(中): 過濾泡沫效應 -> 多樣性指標監控

Step5: 決策建議
  強烈建議執行,投資回報極高,建議:
  - 優先投入資源快速上線
  - 採用成熟開源框架降低開發風險
  - 建立A/B測試平台持續優化
```

### 4.4 評估實作步驟(通用流程)

**AI導入評估八步驟:**
```
Step1: 組建評估團隊
  成員: 業務主管+技術專家+財務分析師+風險管理師

Step2: 界定評估範疇
  明確專案邊界、目標、關鍵假設

Step3: 業務需求訪談
  利害關係人訪談、痛點識別、目標定義

Step4: 技術可行性分析
  資料稽核、技術選型、系統評估

Step5: 財務模型建立
  成本估算、效益量化、ROI/NPV計算

Step6: 風險識別與量化
  風險工作坊、風險矩陣、緩解計畫

Step7: 綜合評估報告
  彙整發現、提出建議、決策支持

Step8: 決策審查會議
  向管理層報告、回答疑問、取得核准
```

### 4.5 常見陷阱與解決方案

**陷阱1:過度樂觀的效益估算**
- 問題:誇大AI效益,忽略實施難度與採用率
- 解決:保守估算+敏感度分析+參考業界標竿

**陷阱2:忽略隱性成本**
- 問題:只計算開發成本,忽略資料工程、變革管理、持續維運
- 解決:完整成本清單+留20%緩衝預算

**陷阱3:資料品質盲點**
- 問題:假設資料可用,實際上存在大量缺失、錯誤、偏差
- 解決:實際資料稽核+PoC驗證

**陷阱4:技術導向而非問題導向**
- 問題:追求最新技術,忽略業務真實需求
- 解決:從問題出發,技術為手段非目的

**陷阱5:孤島式評估**
- 問題:僅技術團隊評估,缺乏業務與財務視角
- 解決:跨職能評估團隊+多維度綜合分析

### 5.2 記憶技巧

**風險矩陣記憶法(紅綠燈):**
```
想像風險評估如過馬路:
綠燈(1-5分): 低風險,安全通過
黃燈(6-12分): 中風險,小心觀察
橙燈(13-20分): 高風險,等待時機
紅燈(21-25分): 極高風險,禁止通行

機率 × 影響 = 風險分數
```

**評估流程記憶法(漏斗模型):**
```
想像篩選專案如淘金:
第一層篩: 業務需求(是否值得做?)
第二層篩: 技術可行(是否做得到?)
第三層篩: 成本效益(是否划算做?)
第四層篩: 風險評估(是否安全做?)
最終產出: 高品質專案(黃金)
```

### 5.3 快速回憶提示

**考試快速回想關鍵字:**
- 評估維度 = 需求+技術+成本+風險+組織
- 財務指標 = ROI + NPV + 回收期
- 風險分數 = 機率 × 影響
- 決策閘門 = Go / No-Go / Defer
- 資料就緒 = 品質 × 數量 × 可及性

### 5.4 易混淆辨析

**ROI vs NPV:**
- **ROI**:投資報酬率,簡單百分比,不考慮時間價值
- **NPV**:淨現值,折現現金流,考慮資金時間價值
- 記憶:ROI是「速食」(快速評估),NPV是「慢燉」(精確分析)

**技術可行 vs 業務可行:**
- **技術可行**:能不能做出來(資料、技術、系統)
- **業務可行**:值不值得做(需求、效益、ROI)
- 記憶:技術問「How」(如何做),業務問「Why」(為何做)

**PoC vs Pilot:**
- **PoC(概念驗證)**:驗證技術可行性,小規模實驗環境
- **Pilot(試點)**:驗證業務價值,真實環境小範圍部署
- 記憶:PoC在「實驗室」,Pilot在「現場」

## 6. 自我驗證 (10%, 200-600字)

### 6.1 選擇題

**題目1**:計算AI專案的ROI時,下列哪項應計入投資成本?
A. 僅軟體開發費用
B. 開發費用+硬體+訓練+維運
C. 僅硬體採購費用
D. 僅人力成本

**答案**:B
**解析**:ROI計算需涵蓋完整成本,包括初期開發(人力、軟體、硬體、訓練、顧問)與年度營運成本(維護、更新、授權)。僅計算部分成本會高估ROI,誤導決策。

---

**題目2**:風險評估矩陣中,風險分數的計算公式是?
A. 機率 + 影響
B. 機率 × 影響
C. 機率 - 影響
D. 機率 / 影響

**答案**:B
**解析**:風險分數 = 機率(1-5) × 影響(1-5),範圍1-25分。此方法能綜合考量風險發生可能性與潛在衝擊,是風險管理的標準做法。

---

**題目3**:下列哪種情況最適合導入AI解決方案?
A. 規則明確且穩定的審批流程
B. 需處理大量影像辨識且規則複雜
C. 資料量小且可用確定性演算法
D. 法規要求100%可解釋性

**答案**:B
**解析**:AI適合處理規則複雜、資料量大、存在模式識別需求的場景。影像辨識符合這些特徵。規則明確穩定的流程用傳統RPA更合適,資料量小或需100%可解釋性的場景不適合AI。

---

**題目4**:淨現值(NPV)計算中的折現率(Discount Rate)通常代表什麼?
A. 銀行存款利率
B. 企業加權平均資本成本(WACC)
C. 通貨膨脹率
D. 專案風險係數

**答案**:B
**解析**:折現率通常使用企業加權平均資本成本(WACC),反映企業融資成本與機會成本。用WACC折現可比較不同專案的真實價值,確保資本配置效率。

---

**題目5**:在AI導入評估中,「組織準備度」主要評估什麼?
A. 伺服器硬體是否充足
B. 組織文化、人才、流程是否支持AI導入
C. AI模型準確度是否達標
D. 競爭對手是否使用AI

**答案**:B
**解析**:組織準備度評估企業在文化(是否擁抱創新)、人才(是否具備AI技能)、流程(是否能配合調整)等面向是否ready for AI。硬體屬技術可行性,模型準確度屬執行階段,競爭對手屬市場分析。

### 6.2 簡答題

**題目6**:請說明為什麼AI專案的成本效益分析比傳統IT專案更複雜?並提出因應方法。

**答案解析**:
- **複雜性原因**:
  1. **效益量化困難**:部分效益(如客戶體驗提升)難以直接換算成金額
  2. **不確定性高**:模型準確度、使用者採用率難以事前精確預測
  3. **隱性成本多**:資料工程、持續訓練、模型維運成本易被低估
  4. **效益遞延**:AI專案可能需較長時間才見效(學習曲線)

- **因應方法**:
  1. **代理指標**:用可量化指標(如響應時間降低%)推估業務價值
  2. **敏感度分析**:設定樂觀/基準/悲觀情境,評估ROI範圍
  3. **PoC驗證**:小規模試點驗證實際效益,降低估算誤差
  4. **分階段投資**:採MVP(最小可行產品)策略,降低初期投資風險
  5. **參考標竿**:研究類似產業案例,校準預期效益

---

**題目7**:在進行技術可行性評估時,資料品質不佳會帶來哪些風險?應如何評估與改善?

**答案解析**:
- **資料品質問題帶來的風險**:
  1. **模型準確度低**:Garbage In, Garbage Out,資料品質直接影響模型效能
  2. **開發時程延長**:需投入大量時間清理資料
  3. **成本超支**:資料工程成本可能遠超預期
  4. **業務價值降低**:模型不準確導致實際效益打折

- **評估方法**:
  1. **資料剖析(Data Profiling)**:檢查缺失值比例、異常值、資料分佈
  2. **資料品質評分**:從完整性、準確性、一致性、及時性四個維度評分
  3. **標註品質檢查**:抽樣檢查標註準確度與一致性
  4. **PoC驗證**:用現有資料訓練模型,驗證實際可達成的準確度

- **改善策略**:
  1. **資料清理計畫**:制定資料清理SOP,投入資料工程資源
  2. **資料增強**:透過合成資料、資料擴增技術補充樣本
  3. **主動學習**:優先標註對模型提升最有幫助的資料
  4. **資料治理強化**:建立長期資料品質管理機制

### 6.3 易錯點提醒

**易錯點1**:將ROI與NPV混淆
- **正確理解**:ROI是簡單的百分比收益率,NPV考慮資金時間價值,兩者互補使用
- **記憶提示**:ROI快速篩選,NPV精確決策

**易錯點2**:認為技術可行就一定值得做
- **正確理解**:技術可行僅是必要條件,還需業務價值、成本效益、風險可控
- **記憶提示**:「能做」≠「該做」,需多維度評估

**易錯點3**:忽略組織準備度
- **正確理解**:技術再好,組織不ready也會失敗,文化、人才、流程同等重要
- **記憶提示**:Technology + People + Process = Success

**易錯點4**:靜態評估,缺乏敏感度分析
- **正確理解**:AI專案不確定性高,應分析關鍵假設變動對結果的影響
- **記憶提示**:樂觀/基準/悲觀三情境,降低決策風險
